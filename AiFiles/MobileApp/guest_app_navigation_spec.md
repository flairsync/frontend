# Guest Mobile App â€” Navigation & Data Spec

Proposed bottom-tab structure for a guest/customer-facing mobile app, and exactly what
backend already supports each tab vs. what's still missing. Endpoints and auth rules
below were read directly from `flairsync-api/src/discovery/discovery.controller.ts`,
`src/user-favorites/user-favorites.controller.ts`, and
`src/jobs/controllers/public-jobs.controller.ts` â€” not inferred from the frontend hooks â€”
so this reflects the real guard on each route, not just what the current web UI happens
to call.

**Auth legend**

| Label | Meaning |
|---|---|
| **Public** | No guard at all. Works identically for anyone. |
| **Optional** | `OptionalJwtAuthGuard`. Works logged-out (often via a guest cookie), but response/behavior differs if a session is present, and some guard logic inside the handler still throws `ForbiddenException` for specific sub-cases (noted per row). |
| **Required** | Hard guard (`JwtCookieGuard`, `JwtCookiePPGuard`, or a manual `if (!userId) throw Forbidden` inside an `Optional`-guarded handler). No account, no access. |

All paths below are relative to `https://api.flairsync.com/api/v1` (the hardcoded prefix every `features/*/service.ts` uses â€” see root `CLAUDE.md`).

---

## Tab bar

```
[ Favorites ]   [ Discover ]   (( QR Scan ))   [ Reservations ]   [ Profile ]
```

Discover is the home/default tab. QR Scan is a center **action button**, not a
navigable screen â€” it opens the camera, decodes, then routes; it has no own tab state.

---

## 1. Discover (home tab)

Search, filter, and browse businesses; view a business's profile, menu, and reviews.
Backed almost entirely by `features/discovery`.

| Action | Method & Path | Auth | Hook |
|---|---|---|---|
| Search/filter businesses (geo, text, type, tags, price, rating) | `GET /discovery/businesses` | Public (Optional guard, but no logged-in-only behavior differs businesses returned are the same; only `isFavorite` flags personalize) | `useDiscoverySearch` / `useInfiniteDiscoverySearch` |
| Business profile | `GET /discovery/businesses/:id` | Public | `useDiscoveryProfile` |
| Business menu | `GET /discovery/businesses/:id/menu` | Public | `useDiscoveryMenu` |
| Floor/table layout (for reservation UI) | `GET /discovery/businesses/:id/tables` | Public | `useDiscoveryTables` / `useDiscoveryFloors` |
| Table availability check | `GET /discovery/businesses/:id/tables/availability` | Public | `useDiscoveryTableAvailability` |
| Reviews list / stats | `GET /discovery/businesses/:id/reviews`, `/reviews/stats` | Public | `useReviews`, `useReviewStats` |
| My review for this business | `GET /discovery/businesses/:id/reviews/mine` | **Required** | `useMyReview` |
| Create/edit/delete review | `POST` / `PATCH` / `DELETE /discovery/businesses/:id/reviews(/:reviewId)` | **Required** | `useCreateReview`, `useUpdateReview`, `useDeleteReview` |
| Favorite toggle (shown inline on cards/profile) | see Favorites tab below | **Required** | `useFavorites` |

**Screens:** Search/results (map or list), Business detail (menu preview, hours,
reviews, "Reserve a table" CTA gated by `business.allowReservations`), Review composer.

**Note:** search itself never requires login â€” a guest can browse the entire catalog
before ever creating an account. Only the account-scoped actions on top of it
(reviewing, favoriting) need one.

## 2. Favorites

Business-level bookmarking only â€” no favoriting of menu items, no collections/lists.

| Action | Method & Path | Auth | Hook |
|---|---|---|---|
| List favorites | `GET /user-favorites` | **Required** (`JwtCookieGuard` on the whole controller) | `useFavorites().useGetFavorites` |
| Add favorite | `POST /user-favorites/:businessId` | **Required** | `useFavorites().addFavorite` |
| Remove favorite | `DELETE /user-favorites/:businessId` | **Required** | `useFavorites().removeFavorite` |

**Screens:** Favorited-businesses grid/list (reuses the same business card as Discover).

**If logged out:** tapping the tab (or the heart icon anywhere) should route to
login/signup â€” there is no guest-favorites fallback, unlike orders/reservations which
have a cookie-based anonymous path. Don't build one client-side; the backend has no
concept of an anonymous favorite.

## 3. QR Scan (center action)

No API call of its own â€” camera capture + client-side decode, then routes based on
payload. `@zxing` is already a project dependency (used today for staff clock-in QR at
`components/management/attendance/ClockInQrDialog.tsx`), so the decode capability
exists; only the guest-facing camera screen needs building.

| Scanned code type | Current backend support | Resulting route |
|---|---|---|
| **Table QR** (printed per-table, generated by owners via `components/qr/*`) | âś… Real, working today | Decodes to `/tbl/:businessId/:tableId` â†’ that page sets the `fs_table` cookie (6h TTL, `utils/cookies.ts`) and hard-redirects to `/diner/:businessId/menu`. **No login required** â€” this is exactly the flow a guest hits today by scanning a physical table code with their phone camera outside the app. |
| **Receipt QR** | âťŚ Doesn't exist â€” no receipt QR payload format, no generation, no backend handler | Would need: a payload spec (likely `businessId` + `orderId`) and a receipt-QR generator on the POS/receipt-printing side first. |
| **Menu QR** | Partially â€” a table QR already lands on the menu; a menu-only QR (e.g. printed on a physical menu, not tied to a table) is a different, undefined payload | Would need its own payload spec; without a `tableId` the diner session has no table context, so ordering/call-waiter would need to degrade to "browse only" or prompt for takeaway. |
| **Ad / marketing QR** | âťŚ Doesn't exist | Would just need a payload spec (e.g. a deep-link URL) â€” this one doesn't need new backend work, just a decided target (business profile? a promo page that also doesn't exist yet, see Gaps). |

**Bottom line:** ship the table-QR case first (it's fully backed end-to-end today,
including the anonymous-guest flow); design the payload schema for the others before
building UI around them so you're not guessing at what the scanner should branch on.

## 4. Reservations

Booking, viewing, cancelling, and the post-booking action timeline. This is where the
guard nuances matter most â€” booking itself is far more open than everything after it.

| Action | Method & Path | Auth | Hook |
|---|---|---|---|
| Book a table | `POST /discovery/businesses/:id/reservations` | **Optional â€” guests can book.** Requires `business.isPublished` and `business.allowReservations`; if logged in, `userId` is attached automatically, otherwise it's created without one. | `useSubmitReservation` |
| List all my reservations (any business) | `GET /discovery/reservations/mine` | **Required** (handler explicitly throws if no `userId`) | `useAllMyReservations` |
| List my reservations for one business | `GET /discovery/businesses/:id/my-reservations` | **Required** | `useMyReservations` |
| Cancel a reservation | `PATCH /discovery/businesses/:id/my-reservations/:reservationId/cancel` | **Required** (also enforces `reservationCancellationWindow`) | `useCancelReservation` |
| Reservation timeline (chat-log style event history) | `GET /discovery/businesses/:id/my-reservations/:reservationId/timeline` | **Required** | `useReservationTimeline` |
| Timeline actions (confirm attendance, running late, request cancel/modify) | `POST /discovery/businesses/:id/my-reservations/:reservationId/action` | **Required** | `useReservationAction` |

**Important asymmetry to design around:** a logged-out guest **can book** a reservation,
but the moment it's made they **cannot see it again** anywhere in-app â€” there's no
guest-reservation cookie equivalent to the guest-order cookie, and every read/cancel/
timeline endpoint hard-requires `userId`. In practice this means:
- If the guest is logged in at the time of booking â†’ it shows up in this tab immediately.
- If not â†’ the booking still succeeds (confirmation should rely on email, not the app),
  but the tab will never surface it retroactively. **Prompt login/signup before or
  right after booking** if you want the reservation to be visible in this tab â€” don't
  let a guest complete a booking believing they can track it here without one.

**Screens:** Booking flow (from a business's Discover page, not this tab), Upcoming/Past
list, Reservation detail with timeline + action buttons.

## 5. Profile

Account, settings, and everything else that doesn't warrant its own tab.

| Section | Method & Path | Auth | Notes |
|---|---|---|---|
| Own profile | `GET /users/me` | **Required** (whole `profile/` route tree is gated by `pages/profile/+guard.ts`, redirects to `/login`) | `useProfile` |
| Update profile | `PATCH /users/me` | **Required** | |
| Account deletion / cancel deletion | `DELETE` / `POST /users/me`, `/cancel-deletion` | **Required** | |
| Data export request/status | `POST /users/me/data-export`, `GET /users/me/data-export/status` | **Required** | |
| My reviews (all businesses) | `GET /discovery/reviews/mine` | **Required** | Could surface here instead of/alongside Discover |
| Job applications | see below | **Required** | Nest here rather than a separate tab |
| Notifications | see Cross-cutting below | **Required** for the list; not a tab | |
| Settings (password, 2FA, preferences) | `features/profileSettings` | **Required** | |

**Jobs (nested under Profile, not a top-level tab):** real, working, but a different
audience/intent (job-seeking) than dining â€” don't give it primary nav real estate.

| Action | Method & Path | Auth |
|---|---|---|
| Browse jobs | `GET /jobs` | Public |
| Job detail | `GET /jobs/:slug` | Public |
| Apply | `POST /jobs/:id/apply` | **Required + professional profile** (`JwtCookiePPGuard` â€” logged in *and* has completed a professional profile) |
| My applications | `GET /jobs/my/applications`, `/my/applications/:applicationId` | **Required + professional profile** |
| Upload/set resume | `POST /jobs/:id/my-application/resume/file`, `PUT .../resume/url` | **Required + professional profile** |

A guest who taps "Apply" without a professional profile needs to be routed into
`features/professionalProfile` profile creation first â€” the guard will otherwise just
403.

## Cross-cutting

**Notifications â€” bell icon, not a tab.** `GET /notifications`, `/unread-count`,
`PATCH /:id/read`, `/read-all` are all logged-in only (`useNotifications` gates every
query on `user != null`). The `NotificationPayload.type` enum is shared with
staff/business-ops notifications (`SHIFT_*`, `TASK_*`, `INVENTORY_LOW_STOCK`, etc.) â€” a
guest app should filter client-side to `RESERVATION | ORDER | PROMO | ALERT | SECURITY`
only, since the backend doesn't filter by audience itself.

**Anonymous-guest session cookies.** Two short-lived (6h) cookies carry a non-account
guest through a single dining visit: `fs_table` (set by the scanned-table landing page)
and `fs_order` (set client-side after `POST /discovery/businesses/:id/orders` succeeds
without a `userId`). These are what let `GET /discovery/businesses/:id/my-orders/:orderId`
resolve "whose order is this" for a logged-out diner (`findOneForGuest` vs.
`findOneForUser` on the backend). There is **no equivalent for reservations or
favorites** â€” see the Reservations section above.

**The existing dine-in session UI stays as-is.** `pages/diner/@businessId/+Layout.tsx`
already has a working, session-scoped 2-tab nav (Menu / My Order) plus an inline Call
Waiter button, entered via table-QR scan. Treat this as a full-screen takeover layered
on top of the main 5-tab bar while a table/order session is active, not something to
fold into the global tabs â€” rebuilding it would duplicate real, tested logic
(`useDinerMode`, `DinerModeStore`, the seated-reservation/active-order polling).

**Call Waiter** (`POST /discovery/businesses/:id/call-waiter`) is **Optional**: logged-in
users can always call; logged-out guests can only if they hold a valid `tableId` for
that business (proof they're physically seated) â€” otherwise 403.

## Gaps â€” don't design a primary tab around these

Confirmed absent from both frontend and backend (grepped `features/`, `components/`,
`models/`; nothing found beyond decorative UI labels):

- **Loyalty / points / rewards** â€” zero backend concept. One unused static "Rewards"
  filter chip in `MarketplaceSidebar.tsx`, nothing behind it.
- **Coupons / gifts** â€” zero backend concept. One decorative stat card on the marketing
  landing page, not a real feature.
- **Referrals / friends** â€” zero hits anywhere.
- **Cross-business order history** â€” `my-orders` only exists scoped to one business
  (`/discovery/businesses/:id/my-orders`); there's no aggregate "all my orders across
  every business I've eaten at" endpoint yet. Fold order history into the Reservations
  tab as "Bookings & Orders" once (if) that aggregation endpoint gets built â€” don't ship
  a global order-history screen against per-business calls fanned out client-side.
- **Marketplace (general/B2B/SaaS)** â€” the guest-facing `pages/marketplace/guest`
  variant works but is scoped to one business's shop items at a time (needs a
  `businessId`); `b2b` and `saas` pages are unwired stubs (`items={[]}`). Not
  tab-worthy yet.

If any of these get prioritized, this doc should be updated with the same
guard-verified-from-controller approach used above before wiring a tab to it.
